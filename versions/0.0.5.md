# Version 0.0.5

Released: 2026-05-22

## Summary

Version 0.0.5 completes the `ConfigTunnel` notice-line feature across the built-in
storage backends. Notice lines are separate text associated with a configuration,
not with individual parameters.

## Added

- `ConfigTunnelFile.readNoticeLines` and `ConfigTunnelFile.writeNoticeLines`.
- `ConfigTunnelJdbc.readNoticeLines` and `ConfigTunnelJdbc.writeNoticeLines`.
- `ConfigTunnelEtcd.readNoticeLines` and `ConfigTunnelEtcd.writeNoticeLines`.
- `ConfigTunnelJdbcBuilder.colNotice(...)`, defaulting to `notice`.
- `ConfigTunnelEtcdBuilder.noticeExtension(...)`, defaulting to `.notice`.
- Test coverage for notice-line reads and writes in file, JDBC, and etcd modules.
- In-memory notice support in `ConfigTunnelFake` for core tests.

## Changed

- File-backed notices are stored in a sibling notice file using:

  ```text
  <localPath><noticeExtension>
  ```

- JDBC-backed notices are stored as one newline-delimited text field on the
  config row, where the parameter name is empty.
- JDBC parameter rows leave the notice field null.
- PostgreSQL and MariaDB table creation now includes the configured notice
  text column.
- etcd-backed notices are stored under a separate key using:

  ```text
  <localPath><noticeExtension>
  ```

- `ConfigTunnelFileTest` now uses `ConfigTunnelFile.builder()` instead of the
  removed direct path constructor.

## Migration Notes

- If you maintain a custom `ConfigTunnel`, implement:

  ```java
  List<String> readNoticeLines(String localPath);
  void writeNoticeLines(String localPath, List<String> lines);
  ```

- If you already have JDBC config tables, add the notice column manually. With
  default settings, the column is:

  ```sql
  notice TEXT
  ```

- For file storage, construct the tunnel with the builder:

  ```java
  ConfigTunnelFile tunnel = ConfigTunnelFile.builder()
    .baseDir(baseDir)
    .noticeExtension(".notice")
    .build();
  ```
